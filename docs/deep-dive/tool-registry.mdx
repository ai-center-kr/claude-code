## 40+ Tools Behind the CLI

What looks like a simple coding assistant from the outside is backed by a comprehensive tool registry with over 40 tools spanning file operations, shell execution, web access, agent orchestration, task management, and internal-only capabilities.

## The Tool Framework

Every tool in Claude Code implements the `Tool` interface defined in `src/Tool.ts`:

```typescript
export type Tool<Input, Output, Progress> = {
  // Identity
  name: string
  isMcp?: boolean
  alwaysLoad?: boolean
  shouldDefer?: boolean

  // Schema (lazy init via getters)
  readonly inputSchema: Input
  readonly outputSchema?: Output

  // Metadata
  description(): Promise<string>
  prompt(): Promise<string>
  userFacingName(input?): string

  // Capability flags
  isEnabled(permissionContext?): boolean
  isConcurrencySafe(input?): boolean
  isReadOnly(input?): boolean
  isDestructive?(input): boolean

  // Execution
  validateInput?(input): Promise<ValidationResult>
  checkPermissions(input, context): Promise<PermissionDecision>
  call(input, context): Promise<ToolResult<Output>>

  // UI rendering (React / Ink)
  renderToolUseMessage(input, options): ReactNode | null
  renderToolResultMessage(output, ...): ReactNode | null
}
```

Tools are constructed via `buildTool()`, which fills in safe defaults (enabled by default, not concurrency-safe, not read-only).

## Tool Registration

Tools are registered via `getAllBaseTools()` and filtered through multiple gates:

```
getAllBaseTools()
  → Filter by feature flags (compile-time)
  → Filter by USER_TYPE (ant vs external)
  → Filter by environment flags
  → Filter by permission deny rules
  → Available tool set
```

<Note>
  Tools with `shouldDefer: true` are not loaded into the initial prompt. They are fetched on-demand via `ToolSearchTool` to keep the prompt size manageable when 40+ tools are available.
</Note>

## Tool Schema Cache

The `toolSchemaCache.ts` module caches JSON schemas for prompt efficiency. Since tool descriptions and schemas are included in every API call, caching prevents redundant serialization:

```typescript
// Cached per tool name, invalidated on tool set changes
toolSchemaCache: Map<string, CachedToolSchema>
```

## Tool Context

Every `call()` and `checkPermissions()` receives a `ToolUseContext` with:

- **Configuration**: available tools, commands, MCP clients, model info
- **Abort signal**: for cancellation propagation
- **App state**: getters and setters for global state
- **File tracking**: `readFileState` map for read-before-write enforcement
- **Permission context**: current mode, rules, and working directories
- **Agent context**: whether this is a subagent, coordinator, or standalone

<Tip>
  The `readFileState` tracking ensures that `FileEditTool` and `FileWriteTool` cannot modify files that have not been read first in the current session -- a safety mechanism that prevents blind overwrites.
</Tip>
