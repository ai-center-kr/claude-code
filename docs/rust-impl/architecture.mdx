# Crate Architecture

The Rust reimplementation is organized as a Cargo workspace with 10 member crates. The architecture follows a strict layered dependency model where higher-level crates depend on lower-level ones, and `cc-core` sits at the foundation.

## Directory Layout

```
src-rust/
├── Cargo.toml                  # Workspace root
└── crates/
    ├── core/       (cc-core)       # Shared types, config, permissions, history, hooks
    ├── api/        (cc-api)        # API client + SSE streaming
    ├── tools/      (cc-tools)      # All tool implementations (33 tools)
    ├── query/      (cc-query)      # Agentic query loop, compact, cron scheduler
    ├── tui/        (cc-tui)        # ratatui terminal UI
    ├── commands/   (cc-commands)   # Slash command implementations
    ├── mcp/        (cc-mcp)        # MCP (Model Context Protocol) client
    ├── bridge/     (cc-bridge)     # Bridge to claude.ai web UI
    ├── cli/        (claude-code)   # Binary entry point
    └── buddy/      (cc-buddy)      # Tamagotchi companion system
```

## Dependency Graph

The dependency flow is strictly acyclic. `cc-core` is the foundation crate that every other crate depends on.

```
                    ┌──────────┐
                    │  cc-cli  │  (binary entry point)
                    └────┬─────┘
                         │
          ┌──────────────┼──────────────┐
          ▼              ▼              ▼
    ┌──────────┐  ┌──────────┐  ┌──────────┐
    │ cc-query │  │  cc-tui  │  │cc-bridge │
    └────┬─────┘  └────┬─────┘  └────┬─────┘
         │             │              │
    ┌────┴────┐        │              │
    ▼         ▼        │              │
┌────────┐┌───────┐   │              │
│cc-tools││cc-api │   │              │
└───┬────┘└───┬───┘   │              │
    │         │        │              │
    ▼         ▼        ▼              ▼
    ┌──────────────────────────────────┐
    │            cc-core               │
    └──────────────────────────────────┘

    ┌──────────┐  ┌──────────┐  ┌──────────┐
    │cc-commands│  │  cc-mcp  │  │ cc-buddy │
    └────┬─────┘  └────┬─────┘  └────┬─────┘
         │             │              │
         ▼             ▼              ▼
         └─────────► cc-core ◄────────┘
```

<Note>
`AgentTool` lives in `cc-query` rather than `cc-tools` to avoid a circular dependency: `cc-tools` would need `cc-query` (for the nested query loop), but `cc-query` already depends on `cc-tools`.
</Note>

## Design Patterns

### Trait-Based Tool System

Every tool implements the `Tool` trait from `cc-tools`. This enables dynamic dispatch, tool discovery, and uniform permission checking:

```rust
#[async_trait]
pub trait Tool: Send + Sync {
    fn name(&self) -> &str;
    fn description(&self) -> &str;
    fn permission_level(&self) -> PermissionLevel;
    fn input_schema(&self) -> Value;
    async fn execute(&self, input: Value, ctx: &ToolContext) -> ToolResult;
}
```

### Streaming SSE Architecture

The API layer uses a producer-consumer pattern: a background tokio task reads the SSE byte stream and emits typed `StreamEvent` values through an `mpsc` channel. The `StreamAccumulator` reconstructs the full message from incremental deltas.

### Permission Model

Four permission modes (`Default`, `AcceptEdits`, `BypassPermissions`, `Plan`) gate tool execution. The `PermissionHandler` trait abstracts over interactive (TUI prompt) and automatic (CI/batch) permission strategies.

### Hook System

Lifecycle hooks (`PreToolUse`, `PostToolUse`, `Stop`, `UserPromptSubmit`, `Notification`) allow external scripts to observe and optionally block tool executions. Hooks are configured via `settings.json` and executed as subprocesses.
