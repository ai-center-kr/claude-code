## Beyond Simple Allow/Deny

Claude Code's permission system in `tools/permissions/` is a multi-layered security architecture that classifies risk, manages approval workflows, prevents path traversal attacks, and even uses an LLM to explain tool risks to the user.

## Permission Modes

The system supports multiple permission modes that control how tool use requests are handled:

| Mode | Behavior |
|------|----------|
| `default` | Interactive prompts -- user approves each action |
| `acceptEdits` | Auto-approve file edits, prompt for shell commands |
| `plan` | Read-only mode, no modifications allowed |
| `auto` | ML-based auto-approval via transcript classifier |
| `bypassPermissions` | Skip all permission checks |
| `dontAsk` | Never prompt the user |
| `bubble` | Bubble permission request to parent agent |

<CodeGroup>
```typescript External Permission Modes
type ExternalPermissionMode =
  | 'acceptEdits'
  | 'bypassPermissions'
  | 'default'
  | 'dontAsk'
  | 'plan'
```

```typescript Internal Permission Modes (includes ML-based)
type InternalPermissionMode =
  | ExternalPermissionMode
  | 'auto'    // Only when TRANSCRIPT_CLASSIFIER enabled
  | 'bubble'  // For subagent permission delegation
type PermissionMode = InternalPermissionMode
```
</CodeGroup>

## Risk Classification

Every tool action is classified into one of three risk levels:

| Level | Description | Example |
|-------|-------------|---------|
| **LOW** | Read-only operations, information gathering | `FileReadTool`, `GlobTool`, `GrepTool` |
| **MEDIUM** | File modifications, controlled writes | `FileEditTool`, `FileWriteTool` |
| **HIGH** | Shell execution, destructive operations | `BashTool` with `rm`, force-push, process killing |

### The YOLO Classifier

Despite its name, the YOLO classifier is a fast ML-based permission decision system. It analyzes the tool use context and decides whether to auto-approve, prompt, or deny. The classifier is backed by the `TRANSCRIPT_CLASSIFIER` feature flag and uses the `auto` permission mode.

The classifier input is generated per-tool:

```typescript
toAutoClassifierInput(input): string
// Each tool provides its own classification input format
```

## Permission Rules

Users can create persistent permission rules that apply across sessions:

```typescript
type ToolPermissionContext = {
  mode: PermissionMode
  additionalWorkingDirectories: Map<string, AdditionalWorkingDirectory>
  alwaysAllowRules: ToolPermissionRulesBySource
  alwaysDenyRules: ToolPermissionRulesBySource
  // ...
}
```

Rules are scoped by source (user settings, project settings, policy settings) and can target specific tool + argument patterns (e.g., "Always allow bash: `git *`", "Always deny bash: `rm -rf *`").

## Protected Files

Certain files are guarded from automatic editing regardless of permission mode:

- `.gitconfig`
- `.bashrc` / `.zshrc`
- `.mcp.json`
- `.claude.json`
- Other shell configuration files

These files require explicit user approval even in `bypassPermissions` mode, as accidental modification could compromise the user's environment.

## Path Traversal Prevention

The permission system actively defends against path traversal attacks:

| Attack Vector | Defense |
|--------------|---------|
| URL-encoded traversals (`%2e%2e`) | Decoded and validated |
| Unicode normalization attacks | NFC-normalized before comparison |
| Backslash injection (`..\\`) | Normalized to forward slashes |
| Case-insensitive manipulation | Case-normalized on case-insensitive filesystems |
| Symlink escape | Resolved before path validation |
| Relative path outside project | Rejected if resolved path exits project root |

The bridge system adds additional validation:

```typescript
// Validates server-provided IDs against /^[a-zA-Z0-9_-]+$/
// Throws on unsafe characters to prevent path traversal
```

<Note>
  The `buildDownloadPath()` function in the file API explicitly returns `null` if a relative path contains `..` components or resolves outside the expected `basePath/sessionId` directory.
</Note>

## Permission Explainer

A separate LLM call generates human-readable risk explanations before the user approves a tool use. When Claude says "this command will modify your git config," that explanation is itself generated by Claude.

The `PermissionExplanation` component manages expand/collapse state for the explainer section, and `PermissionRuleExplanation` shows what persistent rules would be created by "Always allow" or "Always deny" choices.

## The Permission Dialog System

The permission UI is a layered component architecture:

| Component | Purpose |
|-----------|---------|
| `PermissionRequest.tsx` | Top-level dispatcher for tool-specific permission UIs |
| `PermissionDialog.tsx` | Shared visual container (title bar, border, worker badge) |
| `BashPermissionRequest` | Shell-specific with classifier animation, sed detection, sandbox awareness |
| `FilePermissionDialog` | Generic file operation dialog (edit, write, notebook) |
| `WorkerBadge` | Colored badge showing which swarm worker is requesting |
| `PermissionDecisionDebugInfo` | Debug info about decision reasoning (verbose mode) |
| `PermissionPrompt` | Fullscreen layout with sticky footer for plan mode |

## Risky Actions

The system prompt explicitly lists actions that require confirmation:

**Destructive**: deleting files/branches, dropping DB tables, killing processes, `rm -rf`, overwriting uncommitted changes

**Hard-to-reverse**: force-pushing, `git reset --hard`, amending published commits, removing packages, modifying CI/CD

**Shared-state**: pushing code, creating/closing PRs/issues, sending messages (Slack, email, GitHub), posting to external services

<Warning>
  The `bypassPermissions` mode exists but is intended for CI/CD pipelines and automated workflows. It does not bypass protected file guards -- those remain enforced regardless of mode.
</Warning>
